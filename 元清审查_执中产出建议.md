## 二、🟡 需修正

### 2.1 四行色相角的映射不一致

执中在重写任务书 v2.0 §2.3 中设了：

```
ECLIPTIC_ANGLES = {
    "mu": 120.0,   # 春分·木 — 绿
    "huo": 0.0,    # 夏至·火 — 红
    "jin": 210.0,  # 秋分·金 — 白偏蓝
    "shui": 270.0, # 冬至·水 — 深蓝
}
```

但黄道面和 HSL 色相之间的映射存在偏移不一致：

| 节气 | 黄道方位角 | HSL 色相(执中设) | 偏移量 |
|------|-----------|-----------------|--------|
| 春分(木) | 0° | 120° | +120° |
| 夏至(火) | 90° | 0° | -90°（等价于 +270°） |
| 秋分(金) | 180° | 210° | +30° |
| 冬至(水) | 270° | 270° | 0° |

偏移量不一致意味着黄道面角度和 HSL 色相之间**不是统一的线性映射**，而是四个角各自独立设定的。这会导致「黄道面顺行 1° → 色相偏移多少」无法定义。

**建议**：定义一个统一的偏移常量 `Δ_offset`，让黄道面角度全部平移同量映射到 HSL：

```
HSL_H = (黄道方位角 + Δ_offset) % 360

其中 Δ_offset 的选择：
  如果木=绿→120°，则 Δ_offset = 120°（黄道面0°春分→HSL 120°绿）
  则：火=90+120=210°（偏青，不是红）→ 这个结果不对

  如果火=红→0°，则 Δ_offset = 270°（黄道面90°火→HSL 0°红）
  则：木=0+270=270°（蓝，不是绿）→ 也不行

问题可能出在：火=红(0°)、木=绿(120°) 在色相环上不是 90° 差（是120°差），
而天文学的春分→夏至=90°。这意味着黄道面上的均匀 90° 间隔，
在 HSL 色相环上不是均匀分布的——因为四种颜色在光谱中的位置天然不均匀。

两个解决方案：
  A: 承认色相不均匀，保持四行独立标注（当前做法），但要在文档里明确说明
  B: 用色彩科学中的"色温"代替"色相"——春分~5500K(绿白)、夏至~3000K(暖红)、
     秋分~6500K(冷白)、冬至~8000K(蓝)，然后色温→HSL映射由函数计算
```

**建议选 B（色温代替色相）**。因为色温天然对应季节的天光变化，不需要人肉标定角度。但短期内 A 也可以接受——只要文档说清楚不是线性映射。

## 三、🟢 可采纳

### 3.1 四行强度用正弦函数建模（14张图 §4.1）

基于图 14 太阳直射点正弦回归运动，建议将四行强度从独立输入参数变为黄道方位角的函数：

```python
def wuxing_intensity(angle_deg: float) -> dict:
    rad = math.radians(angle_deg)
    return {
        "mu": max(0, math.cos(rad)),      # 春分(0°)最大
        "huo": max(0, math.sin(rad)),     # 夏至(90°)最大
        "jin": max(0, -math.cos(rad)),    # 秋分(180°)最大
        "shui": max(0, -math.sin(rad)),   # 冬至(270°)最大
    }
```

好处：减少四个自由参数，强度由黄道位置自动决定。

### 3.2 土引力类比五带分布（14张图 §4.2）

用地球五带（热带=赤道=中和区，寒带=两极=极端区）类比土引力——越靠近中轴越"土"，越远离越极端。执中的 `TuGravity` 类方向对，算法逻辑与这个物理直觉吻合。

### 3.3 学术格式版作为存证底稿

建议在理论完整（v3.0 双螺旋数学形式化完成）后，将学术格式版作为论文/专利的底稿。当前阶段不要急着公开，只在团队内部迭代。

## 四、🔵 建议调整优先级

### 4.1 v2.0 → v3.0 分两阶段落地

执中在 v3.0 里写了「v2.0 是 v3.0 在 t=0 时刻的快照」——这个判断对。但从工程落地角度，建议：

**阶段一（先交付）**: v2.0 静态球面
- 四行在黄道面，土是明度轴
- 柱内叠加 + 四柱合并 + 三层叠加(T_D/T_L 暂按静态算子)
- 验证色相输出在物理意义上是正确的

**阶段二（后叠加）**: v3.0 螺旋动态
- 在 v2.0 验证通过后，叠加螺旋相位偏移
- 引入时间参数 t，动态色彩场
- 年/月/日/时四柱的多层螺旋嵌套

**理由**：一次跳到动态色场，如果底层算错了后面全歪。分两阶段可以隔离故障。

### 4.2 代码目前只有任务书，需要动手写

重写任务书很详细（6 项逐条修改），但 `叠加规则_v2.0/` 目录下目前只有任务书，没有 `wuxing_sphere_v2.py`。执中需要实际动手重写——任务书再详细，跑不过自检的代码就是纸。

### 4.3 v1.0 火扇区 bug 修复确认

v1.0 的 `"huo": 195.0` 在 HSL 中是青绿色，这个 bug 修复后需要确认 v2.0 代码中 `huo` 的色相值映射正确（见 §2.1 的讨论）。

## 五、对 v3.0 螺旋模型的额外考量

### 5.1 中轴从「北极→南极」改为「巽4→中5→乾6」的影响

v2.0 中轴 = 北极白 ↔ 球心灰 ↔ 南极黑（一条直线）
v3.0 中轴 = 巽4(东南偏上) ↔ 中5(球心) ↔ 乾6(西北偏下)（一条斜线）

这个修正影响所有基于中轴的色彩计算。如果 v2.0 先落地，需要明确中轴的定义是 v2.0 的简化直轴还是 v3.0 的斜轴（建议 v2.0 先用直轴标注"近似"，v3.0 再精确标定斜轴）。

### 5.2 螺旋振幅 α 的标定

v3.0 §3.4 中的参数 α（螺旋振幅，决定动态色彩偏移的强度）是自由参数。需要一个人或机制来标定——是司辰根据命理规律定值，还是用真实穿搭反馈数据回归？目前未明确。

### 5.3 月周期/日周期的层级

v3.0 §5.4 提出了年/月/日/时的四层螺旋嵌套，但考虑到落地可行性，建议第一版只做年螺旋（大运+流年底色），月/日/时螺旋作为未来扩展。一次性嵌套四层螺旋的参数空间太大，无法验证。
